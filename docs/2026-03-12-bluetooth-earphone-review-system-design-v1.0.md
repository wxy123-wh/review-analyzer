# 基于在线评论分析的蓝牙耳机产品改进决策系统
## 标准化软件设计文档 V1.0（V0.1 + V0.2 整合版）

- 输出日期：2026-03-12
- 来源：`魏煌毕设选题摘要.docx`
- 文档定位：毕业设计可答辩 + 工程可落地

## 0. 版本整合说明
1. `V0.1` 提供了完整的软件设计骨架（FR/NFR、测试、风险、里程碑）。
2. `V0.2` 固化了已确认决策（阿里云、第三方聚合 API、工程稳健架构、蓝牙耳机范围）。
3. 本 `V1.0` 将两者合并为单一主文档，后续以本文件为准。

## 1. 已确认关键约束
1. 优先级：工程落地优先。
2. 部署环境：阿里云。
3. 数据接入：V1 接入第三方聚合评论 API。
4. 架构路线：工程稳健型（前后端分离 + 异步分析 + 独立 NLP 服务）。
5. 业务范围：V1 仅蓝牙耳机。

## 2. 项目背景
在线评论是电商产品改进的重要信息源，但评论量大、文本非结构化、人工分析效率低，导致问题识别不系统、改进优先级主观性强。本系统面向产品经理，提供从评论监测到决策验证的闭环能力。

## 3. 目标与边界
### 3.1 项目目标
1. 对蓝牙耳机评论进行结构化分析（主题、方面、情感、问题）。
2. 形成可解释的问题优先级与改进建议。
3. 支持竞品对比与时间趋势分析。
4. 支持改进动作登记与效果验证，形成闭环。

### 3.2 范围
1. 覆盖方面：音质、降噪、续航、佩戴舒适度、连接稳定性、通话质量。
2. 接入第三方聚合评论 API，支持定时增量同步。
3. 提供 Web 看板、查询与报表导出。

### 3.3 非目标
1. 不直接驱动研发工单自动流转。
2. 不做全品类统一模型训练（仅预留扩展机制）。
3. 不做秒级实时流处理（V1 采用分钟级/批处理）。

## 4. 单品类与全品类的设计差异
1. 语义建模：蓝牙耳机可用固定方面本体；全品类需要类目本体与动态配置。
2. 算法架构：单品类可一套规则/模型；全品类需模型路由和类目参数化。
3. 决策可操作性：单品类建议可直接落到具体产品动作；全品类前期更容易停留在通用层。
4. 工程成本：全品类需配置中心、质量监控与更强 MLOps，首版风险显著提高。

## 5. 用户角色与核心场景
### 5.1 角色
1. 产品经理：查看问题优先级、制定改进计划、复盘效果。
2. 市场/运营：跟踪用户关注点、竞品口碑与趋势。
3. 管理者（可选）：查看改进策略与业务影响摘要。

### 5.2 核心场景
1. 导入某型号近 90 天评论，识别高频负面问题。
2. 对比自家与竞品在各方面表现差异。
3. 评估某次改进上线前后口碑变化。

## 6. 功能需求（FR）
1. FR-01 评论接入：第三方聚合 API 定时拉取 + 手工触发补采。
2. FR-02 数据预处理：去重、清洗、归一化、异常文本过滤。
3. FR-03 主题/方面识别：评论主题抽取与方面映射。
4. FR-04 方面级情感分析：输出极性、强度、置信度。
5. FR-05 观点/问题聚类：形成问题簇与代表评论。
6. FR-06 问题优先级排序：按负面率、提及量、增长率、竞品差距综合评分。
7. FR-07 竞品对比分析：同价位/同定位横向比较。
8. FR-08 趋势分析：按日/周/月观察主题热度与情感变化。
9. FR-09 决策看板：输出“问题-影响-建议-证据-优先级”。
10. FR-10 改进验证：登记动作后按窗口期做前后对比。
11. FR-11 报表导出：图表、问题清单、复盘报告。

## 7. 非功能需求（NFR）
1. 性能：10 万条评论离线分析 30 分钟内完成（V1 指标）。
2. 可用性：核心查询接口月可用性 >= 99.5%。
3. 可靠性：任务失败自动重试，支持人工重跑与死信排查。
4. 可解释性：每条建议可追溯到评论证据与评分分项。
5. 安全性：最小权限访问、敏感字段脱敏、密钥托管。
6. 可扩展性：类目配置化扩展（方面词典、权重、模型路由）。

## 8. 系统架构
### 8.1 逻辑分层
1. 前端展示层：可视化看板、查询、导出、动作登记。
2. 后端业务层：认证授权、业务 API、任务编排、策略配置。
3. NLP 分析层：预处理、方面识别、情感分析、聚类、评分。
4. 异步任务层：采集任务、分析任务、重跑任务、告警任务。
5. 数据存储层：结构化业务库 + 评论原文索引库。

### 8.2 阿里云部署映射
1. `ECS/ACK`：部署 API 服务、NLP 服务、Worker。
2. `RDS PostgreSQL`：主业务数据。
3. `Redis`：缓存、队列、限流计数。
4. `OSS`：原始文件、报表、快照归档。
5. `SLS + ARMS`：日志与链路监控告警。
6. `VPC + 安全组 + RAM`：网络隔离与权限治理。

## 9. 数据流（闭环）
1. 评论同步。
2. 预处理清洗。
3. 方面与情感分析。
4. 问题聚类。
5. 优先级评分与建议生成。
6. 看板展示与决策登记。
7. 改进后验证与复盘。

## 10. 关键数据模型
1. `products`：产品基础信息。
2. `competitors`：竞品映射关系。
3. `reviews_raw`：评论原文、来源、时间、评分、匿名作者 ID。
4. `review_aspects`：评论-方面-情感结果。
5. `issue_clusters`：问题簇、关键词、代表评论、严重度。
6. `issue_scores`：评分分项与总分。
7. `improvement_actions`：改进动作与上线信息。
8. `validation_metrics`：改进前后指标对比。
9. `sync_jobs` / `analysis_jobs`：任务状态、耗时、错误。

## 11. 优先级评分机制
`PriorityScore = w1*NegativeRate + w2*MentionVolume + w3*TrendGrowth + w4*CompetitorGap`

- 默认建议权重：`w1=0.35, w2=0.25, w3=0.20, w4=0.20`。
- 支持按业务阶段参数化调整。

## 12. API 草案（REST）
1. `POST /api/v1/sync/start`：启动评论同步任务。
2. `GET /api/v1/sync/jobs/{id}`：同步任务状态。
3. `POST /api/v1/analysis/start`：启动分析任务。
4. `GET /api/v1/analysis/jobs/{id}`：分析任务状态。
5. `GET /api/v1/issues`：问题簇与优先级查询。
6. `GET /api/v1/compare`：竞品对比查询。
7. `GET /api/v1/trends`：趋势分析查询。
8. `POST /api/v1/actions`：登记改进动作。
9. `GET /api/v1/validation`：验证结果查询。

## 13. 推荐技术栈
1. 后端：`Java Spring Boot`。
2. NLP 服务：`Python FastAPI`（独立服务）。
3. 前端：`Vue 3 + TypeScript + Element Plus + ECharts`。
4. 数据：`PostgreSQL + Redis + OSS`。
5. 容器与交付：`Docker`（生产可选 `ACK`）。
6. 监控运维：`SLS + ARMS + 云监控`。

## 14. 测试与验收
1. 单元测试：清洗、分类、评分计算逻辑。
2. 集成测试：同步 -> 分析 -> 展示 -> 验证端到端链路。
3. 回归测试：模型版本/词典更新后的指标稳定性。
4. 业务验收：与人工分析 TopN 问题的一致性比对。
5. 可用性验收：产品经理完成一次闭环流程并可导出报告。

## 15. 风险与应对
1. 第三方 API 不稳定：重试 + 限流 + 死信队列 + 备用批导入。
2. 评论噪声与刷评：垃圾评论过滤、异常账户规则、置信度阈值。
3. 领域偏差：蓝牙耳机词典与人工校准样本持续迭代。
4. 合规风险：采集边界控制、数据脱敏、访问审计。

## 16. 里程碑（建议）
1. M1（2 周）：数据接入与同步链路稳定。
2. M2（2 周）：方面+情感+聚类基线可用。
3. M3（2 周）：看板、评分与导出上线。
4. M4（1 周）：改进动作与效果验证闭环。
5. M5（1 周）：压测、告警、答辩材料沉淀。

## 17. 扩展到全商品类型的演进计划
