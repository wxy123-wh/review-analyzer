# 基于在线评论分析的蓝牙耳机产品改进决策系统
## 标准化软件设计文档 V0.2（已确认关键约束）

- 输出日期：2026-03-12
- 适用场景：毕业设计 + 工程落地导向
- 约束确认：
1. 优先级：工程落地优先
2. 部署环境：阿里云
3. 数据接入：V1 接入第三方聚合评论 API
4. 架构路线：工程稳健型（前后端分离 + 异步分析 + 独立 NLP 服务）
5. 业务范围：V1 仅蓝牙耳机品类

## 1. 目标与范围
### 1.1 目标
1. 建立评论分析到改进验证的闭环：`采集 -> 分析 -> 决策 -> 验证`。
2. 输出可执行问题清单与优先级，降低产品经理人工分析成本。
3. 支持竞品对比与趋势分析，用于产品定位与迭代复盘。

### 1.2 范围
1. 覆盖蓝牙耳机核心方面：音质、降噪、续航、佩戴舒适度、连接稳定性、通话质量。
2. 接入第三方聚合评论 API，支持定时拉取和增量同步。
3. 提供 Web 管理端看板与报表导出。

### 1.3 非范围
1. 不直接对接研发工单系统自动派发任务。
2. 不做全品类统一模型训练（在架构上预留扩展位）。
3. 不做实时流式秒级分析（V1 采用分钟级批处理）。

## 2. 为什么先做蓝牙耳机（与全品类差异）
1. 领域词汇和方面定义稳定，情感与问题识别准确率更高。
2. 决策建议可直接映射到可执行动作（如耳塞规格、降噪算法、麦克风拾音）。
3. 工程复杂度低，可先跑通闭环并形成稳定指标体系。
4. 全品类需要类目本体、模型路由、类目配置中心，首版风险和维护成本明显更高。

## 3. 总体架构
### 3.1 逻辑分层
1. 展示层（Web 前端）：
- 运营看板、问题优先级、竞品对比、趋势分析、改进验证。
2. 应用层（API 网关 + 业务服务）：
- 用户认证、权限控制、任务管理、报表导出、策略配置。
3. 分析层（NLP 服务）：
- 预处理、方面识别、情感分析、问题聚类、优先级评分。
4. 任务层（异步队列）：
- 评论同步任务、分析任务、重跑任务、告警任务。
5. 数据层：
- 关系库存结构化结果，文档检索库存评论原文与索引。

### 3.2 阿里云映射
1. `ECS` 或 `ACK`：部署 API 服务、NLP 服务、任务 Worker。
2. `RDS PostgreSQL`：业务主库（产品、评论元数据、分析结果、动作记录）。
3. `Redis`：任务队列、缓存、限流计数器。
4. `OSS`：原始导入文件、离线报表、快照数据归档。
5. `SLS + ARMS`：日志检索、应用监控、告警。
6. `NAT/VPC/Security Group`：网络隔离与白名单策略。

## 4. 核心功能设计
### 4.1 评论采集
1. 通过第三方 API 拉取评论，支持按产品、时间窗、分页增量同步。
2. 同步策略：日常定时 + 人工触发补采。
3. 失败重试：指数退避，超过阈值进入死信队列并告警。

### 4.2 文本分析
1. 清洗标准化：去重、去噪、文本归一、语言检测、异常值过滤。
2. 方面识别：基于蓝牙耳机领域词典 + 模型分类。
3. 情感分析：输出方面级情感极性与置信度。
4. 问题聚类：将相似负面观点聚合成问题簇，抽取代表评论。

### 4.3 决策支持
1. 优先级评分：
`Score = w1*负面率 + w2*提及量 + w3*增长率 + w4*竞品差距`
2. 建议生成：按问题簇给出“问题描述-影响-建议动作-证据评论”。
3. 验证闭环：登记改进动作后，比较前后窗口指标变化。

### 4.4 报表与权限
1. 支持导出周报/月报（问题 TopN、竞品差距、改进效果）。
2. 角色权限：
- 产品经理：全量查看与动作登记
- 运营/市场：看板与导出
- 管理者：汇总视图

## 5. 数据模型（核心表）
1. `products`：产品基础信息（品牌、型号、价格带、定位）。
2. `competitors`：竞品清单与映射关系。
3. `reviews_raw`：评论原文、来源、时间、评分、作者匿名 ID。
4. `review_aspects`：评论与方面映射，含情感结果与置信度。
5. `issue_clusters`：问题簇、关键词、代表评论、严重度。
6. `issue_scores`：问题簇评分分项与总分。
7. `improvement_actions`：改进动作记录（内容、版本、上线时间、负责人）。
8. `validation_metrics`：改进前后指标对比（负面率、提及量、口碑差距等）。
9. `sync_jobs` / `analysis_jobs`：任务元数据、状态、耗时、错误信息。

## 6. API 设计（V1 草案）
1. `POST /api/v1/sync/start`：启动评论同步任务。
2. `GET /api/v1/sync/jobs/{id}`：查询同步任务状态。
3. `POST /api/v1/analysis/start`：启动分析任务。
4. `GET /api/v1/analysis/jobs/{id}`：查询分析任务状态。
5. `GET /api/v1/issues`：问题簇查询与排序。
6. `GET /api/v1/compare`：竞品对比指标查询。
7. `GET /api/v1/trends`：时间趋势查询。
8. `POST /api/v1/actions`：登记产品改进动作。
9. `GET /api/v1/validation`：改进效果验证结果查询。

## 7. 推荐技术栈（工程落地优先）
### 7.1 后端与任务
1. 后端框架：`Java Spring Boot`（稳定、企业常用、生态成熟）。
2. 任务队列：`Redis + RQ/自研 Worker` 或 `RabbitMQ`（若需要更强可靠投递）。
3. 数据访问：`PostgreSQL + MyBatis/JPA`。
4. API 文档：`OpenAPI/Swagger`。

### 7.2 NLP 服务
1. 服务形态：独立 `Python FastAPI` 服务，供后端调用。
2. 模型方案：先用可解释方案（规则 + 轻量模型），逐步替换更强模型。
3. 领域资产：蓝牙耳机方面词典、同义词库、问题模板库。

### 7.3 前端
1. 框架：`Vue 3 + TypeScript + Element Plus`。
2. 图表：`ECharts`（趋势、竞品对比、问题分布）。
3. 状态管理：`Pinia`。

### 7.4 云上与运维
1. 容器化：`Docker`，生产可选 `ACK`。
2. 日志：`SLS`，按任务 ID 全链路检索。
3. 监控告警：`ARMS + 云监控`，覆盖失败率、耗时、队列堆积。
4. 安全：`RAM` 最小权限、敏感字段脱敏、API Key 密钥托管。

## 8. 非功能指标（V1 建议）
1. 吞吐：单次离线分析支持 10 万条评论（30 分钟内完成）。
2. 可用性：核心查询接口月可用性 >= 99.5%。
3. 可靠性：同步/分析任务失败自动重试，人工可重跑。
4. 可解释性：每个问题簇可回溯到原始评论证据。

## 9. 扩展到全商品类型的演进路径
1. 增加 `category_aspect_config`（类目-方面配置表）。
2. 增加模型路由层：按类目选择词典、特征、模型。
3. 将评分权重按类目参数化，支持类目级策略配置。
4. 加入类目质量监控面板（准确率、覆盖率、漂移告警）。

## 10. 里程碑（工程版）
1. M1（2 周）：数据接入与同步任务稳定化。
2. M2（2 周）：NLP 基线能力（方面+情感+聚类）上线。
3. M3（2 周）：看板、优先级评分与报表导出上线。
4. M4（1 周）：改进动作登记与效果验证闭环。
5. M5（1 周）：压测、监控告警、论文实验材料沉淀。
