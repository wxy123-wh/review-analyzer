# 接口文档

本文只记录当前代码中已经实现的接口。未在代码中确认的行为，一律不写；无法从当前代码确认的内容，会明确标注。

## 1. backend 接口

### 1.1 GET /api/v1/health
- 作用：返回后端健康状态。
- 请求参数：无。
- 返回字段：
  - `status: string`
  - `timestamp: Instant`
- 默认行为：固定返回 `status = "UP"`。
- 异常处理：代码中未定义业务异常分支。

### 1.2 POST /api/v1/sync/start
- 作用：创建同步任务；当 `provider=onebound` 时立即尝试调用 OneBound；其他 provider 按当前 provider/platform 规则落地透明状态。
- 请求体：
  - `provider: string`，必填，非空
  - `platform: string`，可选，为空时后端归一化为 `taobao`
  - `targetProductCode: string`，必填，非空
- 返回字段：
  - `jobId: string`
  - `provider: string`
  - `platform: string`
  - `targetProductCode: string`
  - `status: string`
  - `startedAt: Instant`
  - `fetchedCount: int`
  - `errorMessage: string | null`
  - `analysisHandoffStatus: string | null`
  - `analysisHandoffNote: string | null`
- HTTP 状态：`202 Accepted`
- 默认行为：
  - `provider=onebound`：立即执行一次外部请求并返回 `SUCCEEDED` 或 `FAILED`
  - `provider=aggregator-demo`：保持受控数据第一轨，不触发外部抓取
  - 不支持的 provider 或 platform：返回显式终态，不再以长期 `QUEUED` 伪装
- 边界与异常：
  - `ONEBOUND_API_KEY` 缺失会失败。
  - `ONEBOUND_API_SECRET` 缺失时会回退使用 key。
  - 错误消息会打码 `key=` 与 `secret=`。
  - `analysisHandoffStatus` / `analysisHandoffNote` 只表达后续分析 handoff 准备度，不代表已经自动触发分析。

### 1.3 GET /api/v1/sync/jobs/{id}
- 作用：查询同步任务。
- 路径参数：`id`
- 成功返回：同 `SyncJobResponse`
- 失败返回：`404`，JSON 结构：
  - `message: "sync job not found"`
  - `jobId: string`
- 边界条件：ID 非数字时同样查询不到。

### 1.4 POST /api/v1/analysis/start
- 作用：创建分析任务，并在当前请求内同步执行受控评论分析与结果物化。
- 请求体：
  - `productCode: string`，必填，非空
- 返回字段：
  - `jobId: string`
  - `productCode: string`
  - `status: string`
  - `startedAt: Instant`
  - `finishedAt: Instant | null`
  - `errorMessage: string | null`
- HTTP 状态：`202 Accepted`
- 默认行为：
  - 新任务按 `QUEUED -> RUNNING -> SUCCEEDED/FAILED` 流转。
  - 成功时会把分析结果物化到查询侧使用的分析表。
  - 若 NLP 服务不可用、超时或返回不合法结果，后端会回退到受控规则分析，并以 `status=SUCCEEDED` 返回，同时把降级原因写入 `errorMessage`，前缀为 `degraded:`。
  - 若同一 `productCode` 已存在新鲜的成功任务，且源评论窗口未更新、物化结果仍存在，则直接复用最近成功任务。
- 失败场景：
  - 找不到任何评论时返回 `FAILED`，`errorMessage` 会带 `no reviews found for productCode=...`
  - 其他运行异常也会返回 `FAILED`

### 1.5 GET /api/v1/analysis/jobs/{id}
- 作用：查询分析任务。
- 路径参数：`id`
- 成功返回：同 `AnalysisJobResponse`
- 失败返回：`404`，JSON 结构：
  - `message: "analysis job not found"`
  - `jobId: string`

### 1.6 POST /api/v1/demo-data/init
- 作用：初始化或刷新演示评论数据。
- 请求体：可为空；若传则结构为：
  - `productCode?: string`
- 返回字段：
  - `seedKey: string`
  - `productCode: string`
  - `dataVersion: string`
  - `targetReviewCount: int`
  - `insertedReviewCount: int`
  - `updatedReviewCount: int`
  - `totalReviewCount: int`
  - `durationMs: long`
  - `initializedAt: Instant`
- 默认行为：
  - 请求体缺失或 `productCode` 空白时，回退到 `demo-earphone`
  - 目标评论数固定为 `100`
  - 重复初始化时走 upsert，不会无限新增
- 注意：该接口只写受控原始评论，不会自动补跑分析任务。

### 1.7 GET /api/v1/issues
- 作用：返回问题优先级列表。
- 查询参数：
  - `productCode?: string`，默认 `demo-earphone`
- 返回字段：
  - `items: IssueItem[]`
  - `state: "success" | "empty" | "error"`
  - `notice: string | null`
- `IssueItem` 字段：
  - `issueId: string`
  - `title: string`
  - `aspect: string`
  - `priorityScore: number`
  - `evidenceSummary: string`
- 默认行为：
  - 查询真实物化后的 `issue_clusters + issue_scores`。
  - 未物化且无可用输出时返回 `items=[]`、`state=empty`、`notice` 提示先初始化演示数据。
  - 有物化输出但无负向高优问题时，也返回 `state=empty`，提示当前评论情绪整体平稳。
- 异常处理：聚合失败返回 `items=[]`、`state=error`、`notice="评论洞察正在更新，请稍后刷新重试。"`

### 1.8 GET /api/v1/compare
- 作用：返回主产品与竞品的方面对比分数。
- 查询参数：
  - `productCode?: string`，默认 `demo-earphone`
  - `comparisonProductCode?: string`，前端默认 `demo-earphone-competitor`
- 返回字段：
  - `productCode: string`
  - `comparisonProductCode: string | null`
  - `state: "success" | "missing-target" | "primary-unavailable" | "comparison-unavailable" | "error"`
  - `notice: string | null`
  - `items: CompareItem[]`
- `CompareItem` 字段：
  - `aspect: string`
  - `ourScore: number`
  - `competitorScore: number`
  - `gap: number`
- 默认行为：
  - 查询两侧物化后的方面平均分，不再是静态数据。
  - 当 `comparisonProductCode` 缺失时返回 `state=missing-target`。
  - 当主产品或竞品尚无物化输出时，分别返回 `primary-unavailable` 或 `comparison-unavailable`。
  - 成功时按后端 canonical aspect 集合返回各方面对比项。
- 异常处理：返回 `state=error` 与统一查询失败提示。

### 1.9 GET /api/v1/trends
- 作用：返回趋势点。
- 查询参数：
  - `productCode?: string`，默认 `demo-earphone`
  - `aspect?: string`，默认 `battery`
- 返回字段：
  - `productCode: string`
  - `aspect: string`
  - `points: TrendPoint[]`
  - `state: "success" | "empty" | "error"`
  - `notice: string | null`
- `TrendPoint` 字段：
  - `period: string`
  - `negativeRate: number`
  - `mentionVolume: number`
- 默认行为：
  - 读取物化后的评论方面数据并按 ISO 周聚合。
  - 无数据时返回 `points=[]`、`state=empty`、`notice` 提示先初始化演示数据。
- 异常处理：返回 `points=[]`、`state=error`、统一查询失败提示。

### 1.10 GET /api/v1/wordcloud
- 作用：返回词云数据。
- 查询参数：
  - `productCode?: string`，默认 `demo-earphone`
  - `aspect?: string`，默认 `all`
- 返回字段：
  - `productCode: string`
  - `aspect: string`
  - `items: WordCloudItem[]`
  - `state: "success" | "empty" | "degraded" | "error"`
  - `notice: string | null`
- `WordCloudItem` 字段：
  - `keyword: string`
  - `frequency: number`
  - `weight: number`
  - `sentimentTag: string`
- 默认行为：
  - 基于已物化的评论范围进行词元提取和情绪标记。
  - 最多返回 24 个关键词。
  - 无评论范围时返回 `state=empty`。
  - 有评论但无法提取到可展示关键词时返回 `state=degraded` 与明确 notice。
- 异常处理：返回 `state=error` 与统一查询失败提示。

### 1.11 POST /api/v1/actions
- 作用：登记改进行动。
- 请求体：
  - `productCode: string`，必填，非空
  - `issueId: string`，必填，非空
  - `actionName: string`，必填，非空
  - `actionDesc?: string`
- 返回字段：
  - `actionId: string`
  - `productCode: string`
  - `issueId: string`
  - `actionName: string`
  - `actionDesc?: string`
  - `status: string`
  - `createdAt: Instant`
- HTTP 状态：`201 Created`
- 默认行为：状态固定为 `PLANNED`。
- 隐式逻辑：如果 `productCode` 对应产品不存在，会自动创建产品，且 `brand=unknown`。

### 1.12 GET /api/v1/validation
- 作用：查询动作验证结果。
- 查询参数：
  - `actionId?: string`
- 返回字段：
  - `items: ValidationItem[]`
  - `state: "success" | "empty" | "degraded"`
  - `notice: string | null`
- `ValidationItem` 字段：
  - `actionId: string`
  - `beforeNegativeRate: number`
  - `afterNegativeRate: number`
  - `improvementRate: number`
  - `summary: string`
- 默认行为：
  - 传 `actionId`：只查单个动作。
  - 不传：查全部动作。
  - 优先复用已持久化的 `validation_metrics` 快照；没有快照时，再基于动作上下文与评论窗口现场计算并回写。
- 状态语义：
  - 动作不存在或没有任何动作时返回 `state=empty` 与 notice。
  - 有结果但存在回退总结时返回 `state=degraded`。
  - 样本不足时仍可能返回 `state=success`，但 `summary` 会明确说明暂无法形成稳定结论。

### 1.13 GET /api/v1/showcase/pipeline
- 作用：返回 v1 流水线视图。
- 返回字段：
  - `status: "LIVE" | "DEGRADED" | "RUNTIME_UNAVAILABLE"`
  - `implemented: boolean`
  - `note: string`
  - `stages: ShowcaseStage[]`
- `ShowcaseStage` 字段：`name`、`state`、`detail`
- 默认行为：
  - `implemented=true`
  - 基于最近的 sync、analysis、materialization、actions、validation 持久化状态拼装
  - stage 的 `state` 可能为 `UNAVAILABLE`、`IDLE`、`QUEUED`、`RUNNING`、`SUCCEEDED`、`FAILED`、`DEGRADED`
- 说明：`note` 中的 `v1-state=...`、`strategy=...`、`data-source=...` 是接口自带合同说明，不是前端附加字段。

### 1.14 GET /api/v1/showcase/agent-arena
- 作用：返回 v1 智能体席位视图。
- 返回字段：
  - `status: "LIVE" | "DEGRADED" | "RUNTIME_UNAVAILABLE"`
  - `implemented: boolean`
  - `note: string`
  - `agents: ShowcaseAgentItem[]`
- `ShowcaseAgentItem` 字段：`agentName`、`role`、`state`、`confidence`
- 默认行为：
  - `implemented=true`
  - `agents` 由 `sync-lane`、`analysis-lane`、`insight-lane`、`action-validation-lane` 组成
  - 状态来自真实子系统持久化信号，不是脚本化假 worker

### 1.15 GET /api/v1/showcase/explainability
- 作用：返回问题得分拆解视图。
- 返回字段：
  - `status: "CONTROLLED_DATA_ONLY"`
  - `implemented: boolean`
  - `note: string`
  - `featureContributions: ShowcaseFeatureContribution[]`
- 子项字段：`feature`、`weight`
- 默认行为：
  - `implemented=true`
  - 优先使用当前 top issue 的物化权重拆解；没有可用 top issue 时退回固定权重配置
  - 这是受控数据下的确定性评分解释，不是模型归因或 attribution

### 1.16 GET /api/v1/showcase/chaos
- 作用：返回最近同步、分析、物化链路的韧性运行态信号。
- 返回字段：
  - `status: "STABLE" | "DEGRADED" | "RUNTIME_UNAVAILABLE"`
  - `implemented: boolean`
  - `note: string`
  - `drills: ShowcaseChaosDrill[]`
- 子项字段：`scenario`、`state`、`detail`
- 默认行为：
  - `implemented=true`
  - 条目固定覆盖 `sync-runtime`、`analysis-runtime`、`materialization-runtime`
  - 没有后端运行信号时会返回 `RUNTIME_UNAVAILABLE`
  - `QUEUED` / `RUNNING` / `FAILED` / 物化陈旧都会被映射为需要关注的运行态，而不是脚本化演练文案

### 1.17 POST /api/v1/showcase/reports/preview
- 作用：返回基于当前查询结果拼装的报告预览。
- 请求体：
  - `module: string`，必填，非空
- 返回字段：
  - `status: "LIVE" | "DEGRADED" | "RUNTIME_UNAVAILABLE"`
  - `implemented: boolean`
  - `note: string`
  - `previewSections: string[]`
- 默认行为：
  - `implemented=true`
  - 会读取 `issues`、`compare`、`trends`、`actions`、`validation` 的当前结果拼装段落
  - 若 compare / trends / validation 有任一不为成功态，则整体预览状态降为 `DEGRADED`
  - 当前只提供预览内容，不执行真实导出

## 2. nlp-service 接口

### 2.1 GET /health
- 作用：返回 NLP 服务健康状态。
- 返回字段：
  - `status: string`
  - `timestamp: string`
- 默认行为：固定返回 `status = "UP"`。

### 2.2 POST /analyze
- 作用：执行规则型评论分析。
- 请求体：
  - `jobId: string`
  - `productCode: string`
  - `reviews: string[]`
- 返回字段：
  - `jobId: string`
  - `aspectSentiments: AspectSentiment[]`
  - `issueClusters: IssueCluster[]`
- `AspectSentiment` 字段：
  - `reviewIndex: int`
  - `aspect: string`
  - `polarity: string`
  - `score: float`
  - `confidence: float`
- `IssueCluster` 字段：
  - `aspect: string`
  - `title: string`
  - `mentionCount: int`
- 默认行为：
  - 方面识别完全基于关键词匹配
  - 情感识别完全基于关键词匹配
  - 只有负向方面会进入 `issueClusters`
- 边界条件：
  - 未命中任何方面关键词时返回 `unknown`
  - 未命中正负向词时返回 `NEUTRAL`
- 后端接入说明：backend 会校验 `jobId`、`reviewIndex`、aspect 和 polarity；不合法时不会信任该结果，而是降级回受控规则分析。

## 3. 前端侧调用约定
- `frontend/src/api/client.ts` 为前端唯一集中接口封装。
- 默认后端基地址：`VITE_API_BASE_URL`；缺失时用 `http://localhost:8080`。
- 请求超时：`10000ms`。
- 超时判定：`ECONNABORTED`、HTTP `408`、HTTP `504`。
- 前端测试模式下，大量调用直接使用本地 mock 返回，不访问真实服务。
- 核心查询接口现在都优先消费显式 `state + notice + items/points` 合同，而不是靠空数组猜测含义：
  - `issues` / `actions` / `validation` 映射到 `success`、`empty`、`degraded`、`error`
  - `trends` / `wordcloud` 还会额外映射请求超时为 `timeout`
  - `compare` 保留后端的 `missing-target`、`primary-unavailable`、`comparison-unavailable` 特殊态
- showcase 接口的 `status` 使用后端原始值：`LIVE`、`STABLE`、`DEGRADED`、`CONTROLLED_DATA_ONLY`、`RUNTIME_UNAVAILABLE`；前端只做中文展示映射。
- `implemented` 现在应视为真实实现标志，当前 showcase v1 接口均返回 `true`。

## 4. 无法从当前代码确认
- 无法从当前代码确认 OneBound 第二轨何时会自动接入 analysis lane；当前只能确认同步透明度、原始评论持久化与 handoff 就绪字段。
- 无法从当前代码确认报告预览之外的真实导出链路、权限模型与外部投递系统。
- 无法从当前代码确认生产环境日志落盘、轮转、聚合、监控、告警、备份恢复策略。
