# 基于在线用户评论分析的电商产品改进决策系统（蓝牙耳机）
## 标准化软件设计文档 V0.1（待需求确认）

- 来源文档：`魏煌毕设选题摘要.docx`
- 输出日期：2026-03-12
- 版本：V0.1（根据摘要推导，待需求澄清）

## 1. 项目背景
在线评论是电商产品改进的重要信息源，但评论量大、文本非结构化、人工分析效率低，导致问题识别不系统、改进优先级主观性强。本系统面向产品经理，提供从评论监测到决策验证的闭环能力。

## 2. 项目目标与边界
### 2.1 目标
1. 对蓝牙耳机评论进行结构化分析（主题、方面、情感、问题）。
2. 形成可解释的产品问题优先级与改进建议。
3. 支持竞品对比与时间趋势分析。
4. 跟踪改进后效果，形成闭环验证。

### 2.2 非目标
1. 不覆盖全类目电商商品（V1 仅蓝牙耳机）。
2. 不做自动化产品研发执行，仅提供决策支持。
3. 不替代客服系统与营销投放系统。

## 3. 用户角色与核心场景
### 3.1 角色
1. 产品经理：查看问题优先级、制定改进计划、复盘效果。
2. 市场/运营：跟踪用户关注点、竞品口碑与趋势。
3. 管理者（可选）：查看改进策略与业务影响摘要。

### 3.2 核心场景
1. 导入某型号近 90 天评论，识别高频负面问题。
2. 对比自家与竞品在佩戴舒适度、音质、续航等方面的差异。
3. 评估某次产品改进前后评论情感变化。

## 4. 功能需求（FR）
1. FR-01 评论数据接入：支持文件导入（CSV/Excel）和平台接口接入（可配置）。
2. FR-02 数据预处理：去重、清洗、分词、停用词处理、异常文本过滤。
3. FR-03 主题与方面识别：抽取评论主题并映射到方面标签（如音质、降噪、续航）。
4. FR-04 方面级情感分析：输出方面-情感极性与强度。
5. FR-05 观点/问题聚类：聚合相似问题并输出问题簇摘要。
6. FR-06 问题优先级排序：结合负面率、提及量、增长趋势、竞品差距计算优先级。
7. FR-07 竞品对比分析：支持同价位/同定位产品横向比较。
8. FR-08 时间趋势分析：按日/周/月查看主题热度与情感变化。
9. FR-09 决策看板：输出“问题-原因-建议-优先级”视图。
10. FR-10 效果验证：支持改进动作登记与前后期对比分析。
11. FR-11 报表导出：导出图表、问题清单、复盘报告。

## 5. 非功能需求（NFR）
1. 性能：10 万条评论数据在 10 分钟内完成离线分析（目标值，可调整）。
2. 可用性：核心查询接口可用性 >= 99.5%。
3. 可解释性：每条建议可追溯到评论证据与评分依据。
4. 安全性：用户数据分级访问，敏感字段脱敏存储。
5. 可扩展性：支持新增类目时通过配置扩展方面词典与模型。

## 6. 系统架构设计
采用前后端分离 + 分析任务异步处理架构。

1. 前端应用层：可视化看板、分析查询、报表导出。
2. 后端 API 层：认证、查询接口、任务编排、权限控制。
3. NLP 分析层：主题识别、方面情感、聚类、排序评分。
4. 任务调度层：批处理任务提交、重试、状态跟踪。
5. 数据存储层：业务库（结构化结果）+ 检索库（评论原文）。

## 7. 数据流（闭环）
1. 评论采集/导入 -> 2. 预处理 -> 3. 主题与情感分析 -> 4. 问题聚类 -> 5. 优先级排序 -> 6. 决策建议展示 -> 7. 改进动作登记 -> 8. 效果验证与复盘。

## 8. 关键数据模型（逻辑层）
1. `products`：产品基础信息（品牌、型号、价格带、定位）。
2. `reviews`：评论原文、时间、评分、渠道、产品 ID。
3. `review_aspects`：评论-方面映射与情感结果。
4. `issue_clusters`：问题簇、代表评论、关键词、严重度。
5. `priority_scores`：问题优先级总分与分项分数。
6. `competitor_metrics`：竞品各方面指标快照。
7. `improvement_actions`：改进动作、发布时间、负责人。
8. `validation_results`：改进前后指标对比。

## 9. 优先级评分规则（建议）
`PriorityScore = w1*NegativeRate + w2*MentionVolume + w3*TrendGrowth + w4*CompetitorGap`

- `w1~w4` 在需求确认后配置；默认建议：`0.35/0.25/0.20/0.20`。
- 支持按业务阶段动态调整（新品期、稳定期、清仓期）。

## 10. 接口草案（REST）
1. `POST /api/reviews/import`：导入评论数据。
2. `POST /api/analysis/jobs`：创建分析任务。
3. `GET /api/analysis/jobs/{jobId}`：查询任务状态。
4. `GET /api/issues`：查询问题簇与优先级。
5. `GET /api/compare/competitors`：查询竞品对比。
6. `GET /api/trends`：查询趋势分析。
7. `POST /api/actions`：登记改进动作。
8. `GET /api/validation`：查询改进效果验证结果。

## 11. 部署建议（中立）
1. 开发环境：单机 Docker Compose（前端 + API + DB + 缓存）。
2. 生产环境：容器化部署，分析任务服务可水平扩展。
3. 监控：任务耗时、失败率、接口延迟、模型推理耗时。

## 12. 测试与验收
1. 单元测试：文本清洗、情感分类、评分计算。
2. 集成测试：任务链路（导入到看板）端到端验证。
3. 业务验收：与人工分析结果对比（Top N 问题一致性）。
4. 可用性测试：产品经理完成一次完整“问题识别-动作登记-效果验证”流程。

## 13. 风险与应对
1. 评论噪声高：引入垃圾评论过滤与置信度阈值。
2. 模型领域偏差：建立蓝牙耳机领域词典与人工校准样本。
3. 数据合规风险：限制采集范围并记录来源与授权信息。

## 14. 里程碑（建议）
1. M1（1-2 周）：需求冻结 + 数据样本准备。
2. M2（2-3 周）：分析流水线最小可用版本。
3. M3（2 周）：可视化看板与竞品/趋势模块。
4. M4（1-2 周）：闭环验证与性能优化。
5. M5（1 周）：论文实验与答辩材料沉淀。
