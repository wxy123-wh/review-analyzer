# 代码文档

本文只以当前仓库代码为事实来源，覆盖实际存在的模块、入口、依赖、默认行为、边界条件、异常处理、隐式逻辑、性能与安全注意点。

## 1. 仓库实际结构

### 1.1 backend
- 路径：`backend/`
- 技术栈：Spring Boot 3.3.3、Java 21、spring-web、spring-jdbc、spring-validation、PostgreSQL 驱动、测试用 H2。
- 入口：`backend/src/main/java/com/wh/review/backend/BackendApplication.java`
- 作用：提供健康检查、同步任务、分析任务、演示数据初始化、洞察查询、动作登记、showcase v1 运行态接口。
- 当前主线：受控数据初始化 -> 分析任务同步执行并物化 -> 查询层消费物化结果 -> 前端再用显式状态语义展示。

### 1.2 frontend
- 路径：`frontend/`
- 技术栈：Vue 3、TypeScript、Vite、Vitest、Axios、GSAP、@antv/g2plot。
- 入口：`frontend/src/main.ts`
- 作用：提供单页演示控制台，先经过内部访问门禁，再进入看板模块。

### 1.3 nlp-service
- 路径：`nlp-service/`
- 技术栈：Python 3.11、FastAPI、Uvicorn、Pydantic。
- 入口：`nlp-service/app/main.py`
- 作用：提供基于关键词规则的方面识别、情感识别、打分与聚类。

### 1.4 infra
- 路径：`infra/db/init/001_init.sql`
- 作用：为 PostgreSQL 初始化核心表结构。
- 注意：后端还会在启动时通过 `PersistenceSchemaInitializer` 再次执行 `CREATE TABLE IF NOT EXISTS` 与补列逻辑。

## 2. backend 模块说明

### 2.1 配置与启动模块
- `config/WebConfig.java`：注册全局 CORS，`allowedOriginPatterns("*")`、允许 `GET/POST/PUT/DELETE/OPTIONS`、允许任意请求头、`allowCredentials(true)`。
- `src/main/resources/application.yml`：默认端口 `8080`，OneBound 基础配置来自环境变量。
- 关键注意：
  - CORS 当前仍是宽开放配置。
  - `allowCredentials(true)` 与通配来源同时出现，运行时行为取决于 Spring 对 `allowedOriginPatterns` 的处理；代码未额外收敛来源范围。

### 2.2 控制器模块
- `HealthController`：固定返回 `UP` 与当前时间。
- `SyncController`：创建同步任务、按任务 ID 查询。
- `AnalysisController`：创建分析任务、按任务 ID 查询。
- `DemoDataController`：初始化演示评论数据，请求体可为空。
- `IssueController`：查询问题列表，默认 `productCode=demo-earphone`。
- `CompareController`：查询竞品对比，支持 `comparisonProductCode`，默认主产品 `demo-earphone`。
- `TrendController`：查询趋势，默认 `productCode=demo-earphone`、`aspect=battery`。
- `WordCloudController`：查询词云，默认 `productCode=demo-earphone`、`aspect=all`。
- `ActionController`：登记动作，并提供动作列表/详情读取。
- `ValidationController`：查询验证结果，可选 `actionId`。
- `ShowcaseController`：返回流水线、智能体席位、可解释性、韧性运行态、报告预览等 showcase v1 接口。

### 2.3 服务模块

#### 2.3.1 SyncJobService
- 默认新建任务状态会先持久化，再根据 provider/platform 进入实际终态。
- `provider=onebound` 时会立刻走 OneBound 抓取逻辑。
- 第二轨重点是同步透明度、原始评论持久化与 `analysisHandoffStatus` / `analysisHandoffNote` 准备度，不会自动并入首发主链路。
- 安全注意：错误消息会尝试把 `key=` 与 `secret=` 参数打码。

#### 2.3.2 OneBoundReviewClient
- 调用 OneBound 外部接口并解析评论返回。
- 必须提供 `ONEBOUND_API_KEY`；`ONEBOUND_API_SECRET` 为空时会退回使用 key。
- 请求路径形如 `/{platform}/item_review`。
- 响应计数从若干候选字段中统计数组长度：`items/comments/rateList/list`。
- 边界条件：若响应为空则返回 `0`；若检测到错误结构则抛出异常。

#### 2.3.3 AnalysisJobService
- 不再只是建一条 `QUEUED` 记录，而是承载当前唯一分析物化入口。
- 主流程：
  1. 规范化 `productCode`
  2. 查找是否存在可复用的最近成功任务
  3. 无可复用任务时创建 `QUEUED`
  4. 立刻转 `RUNNING`
  5. 加载受控评论
  6. 调用 `NlpReviewAnalysisClient`
  7. 构建分析结果并调用 `AnalysisMaterializationRepository.replaceOutputs(...)`
  8. 标记 `SUCCEEDED` 或 `FAILED`
- 复用语义：只有当最近成功任务的 `finishedAt` 不早于源评论最新更新时间，且物化输出仍存在时，才会直接复用该任务。
- 降级语义：
  - NLP 调用失败、超时、不可用，或返回不合法 `jobId` / `reviewIndex` / aspect / `polarity` 时，会回退到 `controlledAnalysis(...)`
  - 回退后任务仍标记为 `SUCCEEDED`
  - 降级原因通过 `errorMessage` 传回，格式类似 `degraded:nlp_invalid_response:...`
- 失败语义：
  - 没有评论、物化替换异常等运行错误会标记 `FAILED`
  - 失败信息不会额外包装成新字段，仍写入 `errorMessage`
- 当前事实：analysis 仍是同步请求内执行，不存在独立队列消费者、异步回调或调度器。

#### 2.3.4 NlpReviewAnalysisClient
- 调用 `NLP_BASE_URL + /analyze`，把 `jobId`、`productCode`、评论列表发给 `nlp-service`。
- 对返回结构做合同校验：
  - `jobId` 必须匹配
  - `reviewIndex` 必须落在源评论索引范围内
  - aspect 和 polarity 必须能被后端规范化接受
- 返回结果不是“硬失败或硬成功”二元，而是 `success + optional degradedMessage` 语义，供 `AnalysisJobService` 决定是否回退。

#### 2.3.5 DemoDataInitializationService
- 演示种子键固定：`demo-comments`。
- 数据版本固定：`demo-comments-v1`。
- 默认商品编码：`demo-earphone`。
- 固定目标评论数：`100`。
- 评论时间从 `2026-01-01T00:00:00Z` 起按 6 小时递增。
- 评分按索引周期分布：低分 2.0、中分 3.0、高分 4.5。
- 方面集合固定：`battery`、`bluetooth`、`noise-canceling`、`comfort`、`microphone`。
- 日志会输出 seedKey、productCode、版本、插入数、更新数、总数、耗时。
- 边界条件：请求中的 `productCode` 为空或空白时，会回退到默认商品编码。

#### 2.3.6 DemoReviewAggregationService
- 仍是受控规则分析的基础层，也是 NLP 回退时的事实来源。
- 从 `reviews_raw` 读取评论，再根据评论文本推断方面、根据评分推断情感。
- 方面默认映射：
  - `battery`
  - `bluetooth`
  - `noise-canceling`
  - `comfort`
  - `microphone`
  - 无法识别时为 `unknown`
- 评分情感阈值：
  - `<=2.5` 负向
  - `<=3.5` 中性
  - `>3.5` 正向
- 隐式逻辑：方面归一化支持中英文别名，如“续航/电池”“连接/连接稳定性”“降噪”“佩戴/舒适度”“通话/收音”。

#### 2.3.7 InsightQueryService
- 查询侧现在以 `AnalysisMaterializationRepository` 为主数据源，不再把受控演示聚合直接当长期 compute-on-read 主实现。
- `listIssues`：
  - 读取物化 `issue_clusters + issue_scores`
  - 返回 `IssueListResponse(items, state, notice)`
  - 无物化输出时显式 `empty`
- `compare`：
  - 读取两侧物化后的方面分数
  - 返回 `missing-target`、`primary-unavailable`、`comparison-unavailable`、`success`、`error`
  - 已不再返回静态 compare 数据
- `trends`：
  - 读取物化后的评论方面结果，再按 ISO 周聚合负面率与提及量
  - 返回 `success` / `empty` / `error`
- `wordCloud`：
  - 只在已物化的评论范围内做词元提取
  - 最多返回 24 个关键词
  - 当评论范围存在但提取不到关键词时返回 `degraded`
- `validation`：
  - 优先查持久化的 `validation_metrics`
  - 缺少快照时，按动作时间窗或回退边界现场计算并回写
  - 无动作、动作不存在、样本不足、回退总结都走显式 `state + notice/summary` 语义
- 性能注意：
  - `reviewsCache` 仅用于部分验证回算场景，TTL 为 10 分钟
  - 大多数聚合仍在应用层内存处理，未见批量预计算管线、分页或复杂缓存层

#### 2.3.8 ActionService
- 仍是 `ActionRepository` 的薄封装。
- 支持创建、按 ID 查询、查询全部、补写 launch context，服务于动作列表和验证窗口解析。

#### 2.3.9 ShowcaseService
- 已不是固定演示占位返回，而是基于真实持久化状态和查询输出合成 showcase v1 视图。
- `pipeline()`：
  - 汇总最近 sync、analysis、materialization、actions、validation 状态
  - top-level `status` 为 `LIVE`、`DEGRADED` 或 `RUNTIME_UNAVAILABLE`
  - stage 级别可出现 `UNAVAILABLE`、`IDLE`、`QUEUED`、`RUNNING`、`SUCCEEDED`、`FAILED`、`DEGRADED`
- `agentArena()`：
  - 把真实子系统状态映射到 `sync-lane`、`analysis-lane`、`insight-lane`、`action-validation-lane`
  - `confidence` 取决于状态和是否有 backing data
- `explainability()`：
  - 始终返回 `CONTROLLED_DATA_ONLY`
  - 有 top issue 时展示当前物化权重拆解，否则展示固定权重配置
  - 明确不是模型 attribution
- `chaos()`：
  - 读取最近 sync / analysis / materialization 运行信号
  - 返回 `STABLE`、`DEGRADED`、`RUNTIME_UNAVAILABLE`
  - 这是运行态信号板，不是脚本化故障注入系统
- `reportPreview()`：
  - 汇总 `issues`、`compare`、`trends`、`actions`、`validation` 的当前结果生成预览段落
  - 如果关键子查询不是成功态，则整体标为 `DEGRADED`
- 公共语义：
  - `implemented` 现在固定为 `true`
  - `note` 是合同说明字段，包含 `v1-state`、`strategy`、`data-source`，不只是自由文案

### 2.4 持久化模块

#### 2.4.1 PersistenceSchemaInitializer
- 后端启动后自动执行建表和补列。
- 隐式逻辑：即使 PostgreSQL 已通过 `infra/db/init/001_init.sql` 初始化，应用仍会再次尝试确保表和字段存在。

#### 2.4.2 ProductRepository
- 若 `product_code` 不存在，会自动创建产品。
- 新建时 `product_name=product_code`，`brand=unknown`。
- 边界条件：并发插入时靠唯一键与 `DuplicateKeyException` 重试兜底。

#### 2.4.3 DemoReviewSeedRepository
- 演示评论为“先更新，后插入，冲突再更新”的 upsert 行为。
- 同时维护 `demo_seed_versions`。
- 幂等性：同一 `seed_key + product_code` 重复初始化不会无限新增评论，而是转为更新。

#### 2.4.4 AnalysisMaterializationRepository
- 当前查询主链路依赖的核心仓储。
- 负责：
  - 覆盖式写入 `review_aspects`、`issue_clusters`、`issue_scores`
  - 查询 issue、compare、trend、wordcloud 所需的物化视图
  - 判断某产品是否已有物化输出
  - 查询最新源评论更新时间
  - 查询 top issue 的权重拆解
- 语义重点：compare、issues、trends、wordcloud、部分 showcase 都以这里的物化状态为真相来源。

#### 2.4.5 DemoReviewQueryRepository
- 仅按 `productCode` 查询评论，按时间和 ID 升序返回。

#### 2.4.6 ActionRepository
- `status` 统一持久化为 `PLANNED`。
- 如果时间字段为空，读取时会用 `Instant.now()` 兜底。
- 现在还承担验证上下文和 launch window 的部分读写支撑。

#### 2.4.7 ValidationMetricsRepository
- 持久化动作验证快照。
- 验证接口优先读取这里的最新快照，避免每次都重新切窗计算。

#### 2.4.8 SyncJobRepository / AnalysisJobRepository
- 两类任务均以自增主键作为 `jobId`。
- `AnalysisJobRepository` 现在支持 `markRunning`、`markSucceeded`、`markFailed` 与按产品查最近成功任务。
- 查询不存在或 ID 非数字时返回空，由控制器转换为 `404` 或空结果。

## 3. frontend 模块说明

### 3.1 启动与全局入口
- `src/main.ts`：挂载 `App.vue`。
- `src/api/client.ts`：集中封装所有接口访问，超时时间 `10000ms`。
- `src/types/domain.ts`：定义前端领域模型与状态联合类型。
- `src/utils/showcaseCopy.ts`：把 showcase 原始状态归一化为前端展示文案，重点覆盖 `LIVE`、`STABLE`、`DEGRADED`、`CONTROLLED_DATA_ONLY`、`RUNTIME_UNAVAILABLE`。

### 3.2 应用壳层 App.vue
- 未登录时显示 `LoginGate`。
- 登录参数不是组件内硬编码，而是来自：
  - `VITE_INTERNAL_ACCESS_USERNAME`
  - `VITE_INTERNAL_ACCESS_PASSWORD`
  - `VITE_INTERNAL_ACCESS_DISPLAY_NAME`
  - `VITE_INTERNAL_ACCESS_HINT`
- 登录后显示窄侧边栏与内容区。
- 默认模块：`overview`。
- 已接入模块：
  - `overview`
  - `issues`
  - `compare`
  - `trends`
  - `wordcloud`
  - `actions`
  - `validation`
  - `showcase-pipeline`
  - `showcase-agent-arena`
  - `showcase-explainability`
  - `showcase-chaos`
  - `showcase-report-center`
- 模块合同清单现在内嵌在 `moduleContracts`：
  - `issues`、`compare`、`trends`、`wordcloud`、`actions`、`validation` 标记为 `real`
  - `overview`、`showcase-explainability` 标记为 `controlled-data-only`
  - `showcase-chaos` 标记为 `real`，但策略为 `hide-by-default`
- `VITE_SHOW_CHAOS_MODULE` 是 chaos 模块唯一开关，关闭时模块会进入 hidden list，而不是彻底消失无痕。
- 登录成功后会并行加载：后端健康、问题、动作、对比、趋势、验证。
- `wordcloud` 和 showcase 数据按需加载。
- `ensureModuleData()` 在进入任一 showcase 模块时会并行预加载全部 showcase 接口。

### 3.3 状态处理与前端合同语义
- `resolveContractMessage(...)` 统一把 `empty`、`degraded`、`runtime-unavailable`、`disabled`、`timeout`、`error` 映射成前端提示。
- `applyIssueResponse`、`applyActionResponse`、`applyValidationResponse`、`applyTrendResponse`、`applyCompareResponse`、`applyWordCloudResponse` 把后端合同对象落到页面状态。
- `overview` 不是单独接口，而是客户端根据 issues/actions/validation 三块状态拼出来的受控数据总览：
  - 全部还在 idle/loading 时显示 loading
  - 任一子块 degraded/error/runtime-unavailable/disabled 时，总览状态也转为 degraded
  - 三块都为空时，总览显示 empty
- `compare` 现在单独保留：
  - `missing-target`
  - `comparison-unavailable`
  - `primary-unavailable`
  - `error`
  这些不是通用 collection state。

### 3.4 登录与交互模块

#### 3.4.1 LoginGate.vue
- 校验规则：账号长度至少 3，密码长度至少 6。
- 提交后固定等待 800ms 再判定。
- 成功凭证来自 `App.vue` 传入的 env-backed props。
- 安全注意：这是前端内部访问门禁，不具备后端鉴权能力。

#### 3.4.2 AnimatedCharacters.vue
- 依赖 GSAP。
- 提供 4 个互动角色：`purple`、`black`、`orange`、`yellow`。
- 交互能力：
  - 鼠标跟随
  - 单击跳跃
  - 双击高亮
  - 密码显示/隐藏联动
- 性能注意：组件内使用 `mousemove`、`requestAnimationFrame`、多个定时器和 GSAP `quickTo`；代码未见节流配置之外的进一步性能保护。

### 3.5 业务展示模块
- `StatusCard.vue`：状态卡片。
- `IssueTable.vue`：问题清单，消费显式 `state + message`。
- `CompareTable.vue`：竞品对比，消费 compare 特殊态与提示。
- `ActionList.vue`：展示动作列表，并触发“登记演示数据动作”。
- `ValidationList.vue`：展示验证结果与改善率。

### 3.6 图表模块

#### 3.6.1 TrendList.vue
- 成功时优先渲染 G2Plot 折线图；测试模式下改为列表。
- 失败、超时、空数据三种状态均有独立提示与重试按钮。
- `runtime-unavailable` 与 `disabled` 也能沿同一合同路径展示。
- 点击图上点位可查看周期、负面率、提及量详情。

#### 3.6.2 WordCloudPanel.vue
- 成功时优先渲染 G2Plot 词云；测试模式下改为标签列表。
- 支持 `POSITIVE/NEGATIVE/NEUTRAL` 三种词性配色，未知值回退为 `NEUTRAL`。
- 展示当前高频词、notice 与重试状态。
- 只有在 `state=success` 时才展示后端 notice 文案；其他状态使用前端标准消息。

### 3.7 Showcase 展示模块
- `ShowcasePipelinePanel.vue`：展示流水线 stage 列表与顶层状态。
- `ShowcaseAgentArenaPanel.vue`：展示智能体席位与 confidence。
- `ShowcaseExplainabilityPanel.vue`：展示确定性得分拆解权重。
- `ShowcaseChaosPanel.vue`：展示运行态告警与降级信号。
- `ShowcaseReportCenter.vue`：选择目标模块并请求报告预览。
- 共同特点：
  - 都依赖后端真实 showcase v1 返回
  - `implemented` 当前应为 `true`
  - `status` 的中文文案通过 `showcaseCopy.ts` 统一映射，不改变原始合同值

### 3.8 前端 API 客户端默认行为
- 测试模式下大量接口直接返回内置受控数据，不发真实请求。
- 非测试模式下：
  - `fetchIssues` / `fetchActions` / `fetchValidation` 根据后端显式 `state`，或在缺失时按 `items + notice` 推导状态
  - `fetchTrends` / `fetchWordCloud` 除后端显式状态外，还会把 Axios timeout 归一化成 `timeout`
  - `fetchCompare` 保留 compare 专属状态集合
  - showcase 接口失败时不再伪装成未实现，而是回退到 `implemented=true + status=RUNTIME_UNAVAILABLE`
- `DEFAULT_COMPARE_PRODUCT_CODE = 'demo-earphone-competitor'`，因此 compare 的默认竞品并非空值。

### 3.9 前端维护脚本
- `scripts/check-banned-copy-terms.mjs`：扫描 `frontend/src` 与仓库根 `README.md`。
- 命中禁用词后脚本会退出非零，导致 `npm test` 失败。

## 4. nlp-service 模块说明

### 4.1 main.py
- 定义 FastAPI 应用 `wh-nlp-service`。
- 提供 `/health` 与 `/analyze`。

### 4.2 analyzer.py
- 规则型实现，不加载模型、不访问外部服务。
- 方面关键词：
  - `battery`
  - `bluetooth`（关键词里兼容 `connectivity`）
  - `noise-canceling`（关键词里兼容 `noise_canceling`）
  - `comfort`
  - `microphone`（关键词里兼容 `call quality` / `call_quality`）
- 情感关键词：
  - 负向：`差/断开/噪音/卡顿/不稳/发热/一般/掉电/衰减`
  - 正向：`好/稳定/清晰/舒适/流畅/满意`
- 打分固定：负向 `-0.78`，正向 `0.82`，中性 `0`。
- 置信度规则：文本长度 `<8` 返回 `0.74`，否则 `0.88`。
- 聚类逻辑：只对负向方面做计数聚类，并按提及次数降序排序。
- 与 backend 的关系：backend 接受这些 NLP 输出，但会把 legacy aspect 名通过别名映射到 canonical aspect 集合；若仍无法映射，则直接降级回受控规则分析。

### 4.3 schemas.py
- `AnalyzeRequest`：`jobId`、`productCode`、`reviews`。
- `AnalyzeResponse`：`jobId`、`aspectSentiments`、`issueClusters`。

## 5. 数据与初始化顺序
- 开发/生产 compose 都先起 PostgreSQL，再起 Redis 与 NLP，再起 backend，最后起 frontend。
- PostgreSQL 初始化来源有两处：
  1. 容器启动挂载的 `infra/db/init/001_init.sql`
  2. backend 启动时的 `PersistenceSchemaInitializer`
- 当前首发主顺序是：
  1. `POST /api/v1/demo-data/init`
  2. `POST /api/v1/analysis/start`
  3. 前端读取 issues / compare / trends / wordcloud / validation / showcase
- 外部来源第二轨可额外产生 `sync_jobs` 与 `reviews_raw` 数据，但不自动替代上述主顺序。

## 6. 关键限制与“无法从当前代码确认”
- 外部来源 / OneBound 仍是第二轨骨架，只能确认同步透明度、原始评论落库与 handoff 准备字段，无法确认何时会自动接入 analysis lane。
- analysis 任务虽然已同步执行并物化，但当前没有独立异步调度器、批处理编排器或自动重试器。
- 可解释性当前明确只是 `CONTROLLED_DATA_ONLY` 的固定权重拆解，不能当作模型归因系统。
- chaos 模块是真实运行态信号板，但前端是否显示仍受 `VITE_SHOW_CHAOS_MODULE` 控制；关闭时是产品策略隐藏，不是后端未实现。
- 报告中心当前只确认预览拼装，不确认真实导出、权限审批、外部投递链路。
- 无法从当前代码确认生产环境日志落盘、轮转、聚合、监控、告警、备份恢复策略。
