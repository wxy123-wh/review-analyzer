# 演示脚本与答辩口径（T8-9.4）

## 1. 演示目标

用“评论数据 -> 问题 -> 趋势图 -> 词云 -> 动作 -> 验证”的闭环，在 6-8 分钟内讲清系统当前可交付价值与边界。

## 2. 演示前准备（建议）

1. 启动服务：`docker compose up --build`
2. 初始化演示评论：调用 `POST /api/v1/demo-data/init`
3. 打开前端：`http://localhost:5175`
4. 登录账号：`wxy / 123456`

## 3. 演示脚本（主线）

1. 登录页（约 1 分钟）
- 先输入错误密码触发失败文案与怪物摇头。
- 再输入 `wxy / 123456` 成功进入看板。
- 强调三怪物交互（跟随、转头、点击蹦跳）是“可解释交互”，不是纯装饰。

2. 问题与趋势（约 2 分钟）
- 进入“问题”说明高优先级问题来自演示评论聚合，不是硬编码标题。
- 切到“趋势图”，说明时间窗口下的负面率与提及量变化。

3. 词云（约 1.5 分钟）
- 展示高频关键词权重与情感编码（正/中/负）。
- 强调词云与趋势都由同一批演示评论派生，口径一致。

4. 动作与验证（约 2 分钟）
- 在“动作”模块登记一条改进行动。
- 切到“验证”说明前后负面率变化摘要，形成闭环。

5. 收口（约 1 分钟）
- 强调当前为“演示数据版本（demo-comments-v1）”。
- 说明已具备可复现初始化、回归命令和验收记录，可用于评审与阶段验收。

## 4. 问题答辩口径

### Q1: 数据是真实线上数据吗？
- 口径：不是。当前是可复现演示数据，目标是保证链路稳定、可回放、可追溯；真实数据接入是下一阶段。

### Q2: 为什么固定 100 条？
- 口径：100 条是演示与回归平衡点，既能覆盖趋势/词云/验证链路，又能控制初始化和测试时长。

### Q3: 为什么登录账号固定为 `wxy / 123456`？
- 口径：这是演示环境的显式约束，降低现场变量；不代表生产鉴权方案。

### Q4: 混沌演练为什么没放在默认主线？
- 口径（处理结论）：默认隐藏。项目经理演示模式聚焦评论决策主线，弱相关演练不默认展示。
- 补充：若需要展示，可通过 `VITE_SHOW_CHAOS_MODULE=true` 打开“韧性演练”，并说明其对“评论同步/分析/看板更新”的链路映射。

### Q5: 词云和趋势是否会口径冲突？
- 口径：不会。二者都来自同一演示评论集合，趋势看时间变化，词云看关键词权重，维度不同但数据源一致。

## 5. 演示禁忌

1. 不把演示结论直接解释为真实市场结论。
2. 不承诺“当前就是生产级能力”。
3. 不在现场切换未验证配置（除非提前准备了回滚方案）。
