## Context

当前仓库已经具备可运行的前后端与 NLP 骨架，但在“产品经理演示”语境下存在四类关键问题：
- 语言与文案不统一：中英混杂，且含“炫技/占位/测试模式”等非业务词。
- 数据闭环不足：后端核心洞察多为硬编码，尚未由评论数据稳定驱动。
- 可视化表达弱：趋势为纯文字，无词云图，阅读门槛高。
- 登录体验与业务目标偏离：当前为“非空即通过 + 单机器人”。

本次设计面向“项目经理可理解、可复现、可演示”的目标，要求以最小范围重构形成一条完整闭环：
演示评论数据 -> 分析指标 -> 图形化呈现 -> 改进动作 -> 验证结果。

## Goals / Non-Goals

**Goals:**
- 全前端展示文案中文化，并建立文案禁用词规约（禁止“炫技、PLACEHOLDER、测试模式”等）。
- 构建可复现的 100 条演示评论数据源，并让问题、趋势、词云、动作、验证围绕该数据工作。
- 趋势图与词云图上线到主导航，替代纯文字趋势表达。
- 登录页完成三怪物交互改造（红蓝黄三种形态）并接入固定账号密码校验（`wxy` / `123456`）。
- 收敛弱相关模块，确保导航项都能解释其与评论决策链路的关系。

**Non-Goals:**
- 不引入生产级鉴权体系（JWT、RBAC、审计登录等）。
- 不在本次实现真实分布式任务编排与真实混沌注入平台。
- 不构建复杂 BI 平台，仅提供项目经理评审可用的关键图表能力。

## Decisions

1. **采用“演示数据模式”替代“测试模式”语义**
- 决策：保留“可离线演示”的技术路线，但将命名、文案与接口说明统一为“演示数据”。
- 原因：满足评审语境，降低“这是测试用假数据”带来的认知负担。
- 备选方案：彻底移除前端本地兜底，仅依赖后端在线接口。  
  放弃原因：现场演示环境不稳定时风险高。

2. **以数据库中的评论数据作为主驱动，避免纯硬编码洞察**
- 决策：新增或启用演示评论表数据（100 条）并提供确定性 seed 入口；洞察计算优先读取该数据。
- 原因：可解释、可回放、可追溯，且能支撑词云统计与趋势聚合。
- 备选方案：继续在 `InsightQueryService` 内硬编码数组。  
  放弃原因：无法支撑词云/趋势扩展，也无法回答“数据从哪来”。

3. **趋势与词云使用前端图形库统一实现**
- 决策：在前端引入一个轻量图表库，同时实现折线/柱状趋势与词云渲染。
- 原因：减少自绘成本，确保移动端可用与交互一致性。
- 备选方案：仅用 CSS/HTML 自绘。  
  放弃原因：维护成本高，功能边界受限。

4. **登录怪物交互采用状态机建模**
- 决策：将怪物行为拆分为 `tracking / typing / error / jumping` 状态，并支持三实例配置（红蓝黄 + 形态差异）。
- 原因：避免条件分支失控，便于测试与后续迭代。
- 备选方案：直接在模板里写条件 class 拼接。  
  放弃原因：交互增多后可维护性差。

5. **模块治理采取“双策略”：默认收敛 + 可解释保留**
- 决策：默认隐藏与评论链路弱相关模块（如“混沌演练”）；如保留则改为“评论链路韧性演练”并给出清晰映射。
- 原因：符合项目经理场景，减少“看起来很炫但无关业务”的质疑。
- 备选方案：原样保留展示模块。  
  放弃原因：与用户反馈冲突，且影响验收聚焦。

## Risks / Trade-offs

- [Risk] 新增演示数据与现有初始化逻辑并存导致表结构漂移  
  -> Mitigation: 统一 `infra/db/init/001_init.sql` 与运行时 schema initializer；新增启动校验日志。

- [Risk] 图表库接入引入包体积增长  
  -> Mitigation: 仅按需引入模块；设置首屏 lazy-load（词云可延后渲染）。

- [Risk] 登录交互动画过多影响低性能设备  
  -> Mitigation: 提供 `prefers-reduced-motion` 与移动端降级策略（关闭粒子与部分动画）。

- [Risk] 收敛模块后与既有测试断言冲突  
  -> Mitigation: 先更新 OpenSpec 任务，再同步改写前后端测试断言并运行全量测试。

- [Risk] 三怪物交互行为描述存在实现歧义  
  -> Mitigation: 在 specs 中为每个行为定义明确场景（WHEN/THEN）与验收标准。

## Migration Plan

1. **阶段 A：规范与清单**
- 完成 OpenSpec proposal/design/specs/tasks 并通过 `openspec validate`。

2. **阶段 B：数据与接口**
- 补齐 100 条演示评论 seed。
- 将 issues/trends/wordcloud 数据改为从演示评论聚合输出。
- 增加词云接口与必要 DTO。

3. **阶段 C：前端体验**
- 文案中文化与禁用词清理。
- 趋势图替换与词云模块接入。
- 登录三怪物重构与账号校验。
- 模块导航收敛（混沌演练处理策略落地）。

4. **阶段 D：测试与验收**
- 更新前端 Vitest、后端 MockMvc、NLP pytest 断言。
- 跑通 `frontend test` / `backend test` / `nlp tests`。
- 输出“项目经理演示脚本 + 验收截图清单”。

5. **回滚策略**
- 保留原模块开关（feature flag）与旧登录组件分支，在单个 release 内可快速回退。

## Open Questions

- “混沌演练”最终是默认隐藏，还是保留但重命名为“评论链路韧性演练”？
- 100 条演示评论是否需要覆盖多语言/多平台来源，还是先聚焦中文电商场景？
- 词云是否要求支持“按方面筛选”（电池/连接/通话）作为首版能力？
- 登录失败摇头动画触发次数与节流策略（每次失败都触发，还是短时间内限次）？
